# Dataset Reviewer

Скилл-**ревьюер**: критикует валидационный датасет **и процесс его сборки**, опираясь на принципы
датасет-менеджмента как на источник истины. Парный к [`dataset-builder`](../dataset-builder/SKILL.md):
builder **собирает**, reviewer **разбирает**. Reviewer ничего не правит за участника — он даёт
обратную связь и оставляет решение человеку (это согласуется с тем, что зачёт даёт **суждение**
участника, а не сам артефакт).

> **Раздаётся бандлом из трёх скилов** (всегда вместе): [`dataset-builder`](../dataset-builder/SKILL.md)
> (содержит общие справки `references/`), `dataset-reviewer` и
> [`schema-guided-reasoning`](../schema-guided-reasoning/SKILL.md). Reviewer берёт принципы из
> `references/` соседнего builder'а — копия не дублируется.

## Главное правило: ревьюим процесс, а не только файл

Финальный датасет — лишь верхушка. **Многие решения в нём не видны**: почему выбрали именно эти
способности, почему такой путь (извлечение/синтез/гибрид), как стратифицировали, какие техники и
скрипты применяли, что осознанно отсекли, как проектировали метрику и валидировали. Поэтому reviewer
оценивает **всю траекторию работы** участника по `dataset-builder` — план-документ, ключевые решения,
шаги, скрипты — **и** итоговый датасет. Если виден только финальный файл — так и пиши: «процесс не
предъявлен, часть осей оценить нельзя».

## Источник истины — принципы датасет-менеджмента

Reviewer **не выдумывает** критерии из головы. Перед ревью он **проходит по справкам** и извлекает
из них принципы, по которым оценивать (те же справки, на которые опирается `dataset-builder`, — ревью
и сборка говорят на одном языке):
[`../dataset-builder/references/`](../dataset-builder/references/) —
[обзор и цикл](../dataset-builder/references/overview.md),
[анализ](../dataset-builder/references/analysis.md),
[получение](../dataset-builder/references/synthesis.md),
[метрики](../dataset-builder/references/metrics.md),
[управление](../dataset-builder/references/management.md).

> Если скил используется внутри учебного репозитория курса — там есть **развёрнутый разбор с
> worked-example** (`doc/materials/dataset-management/`). Справки `references/` — самодостаточная выжимка
> из него; жёсткой зависимости от репозитория курса нет.

## Что подать на вход

1. **Справки с принципами** (`references/` выше) — чтобы знать, по каким принципам ревьюить.
2. **Полную траекторию работы участника** по `dataset-builder` — план-документ / лог решений / шаги /
   скрипты (как собирал).
3. **Сам датасет** (финальные записи) и, если есть, базу знаний / постановку «что проверяем».

Чем полнее траектория — тем глубже ревью. Минимум — сам датасет (тогда честно отметь ограниченность разбора).

## Что reviewer проверяет (оси)

Сначала — извлечённые из справок принципы; ниже — как они ложатся на ревью. **Domain-agnostic ядро**:
фундаментальные паттерны обязательны для любого домена; наши конкретные типы (easy / multi-chunk /
negative / clarifying / multi-turn …) — **лишь примеры**, не требование (у участника типы свои).

**Оси процесса (как шёл):**
- **Что проверяем.** Сформулирована **способность** системы, а не «набор вопросов»? Приоритизирована?
- **Путь и источник.** Извлечение / синтез / гибрид — выбран осознанно и уместно под наличие данных?
- **Стратификация и покрытие.** Есть осмысленная единица покрытия? Редкие категории не потеряны?
- **Техники.** Где уместно — персоны/перефразирование (разнообразие), SGR/structured output (форма),
  негативные инструкции, фиксированный seed, скользящее окно, скрипты анализа большой моделью?
- **Метрика — вместе с датасетом.** Метрика **вытекает** из того, что проверяем, и из формы ground truth?
  Если ground truth неудобен — обоснован ли reference-free / trajectory-подход?
- **Осознанные отсечения.** Что вынесено за скоуп — помечено явно (а не молча пропущено)?

**Оси датасета (что получилось):**
- **Репрезентативность.** Примеры похожи на реальные запросы/боли, а не на шаблон?
- **Покрытие способностей.** Все заявленные способности/измерения представлены? Нет перекоса/дублей?
- **Форма записи.** Консистентный формат (напр. ChatML input/expected_output + metadata)? Поля на месте?
- **Корректность эталона.** Ground truth верен; на реальных данных — не выдуман (реплики/факты дословны)?
- **Валидация.** Проведена структурная / семантическая / Validation Sample? Был ли прогон сильной моделью?
- **Управление.** Обезличивание реальных данных; иммутабельная версия; где живут бинарные вложения.
- **Анти-паттерны.** Шаблонность, дубли, утечки PII, выдуманные факты, «галочное» покрытие.

## Как работает

1. **Прочитать справки** → выписать применимые принципы (коротко, для себя).
2. **Изучить траекторию + датасет** участника.
3. **Пройтись по осям** — по каждой: что **норм** / что **не норм или не продумано** / **конкретная
   корректировка**. Ссылайся на принцип из справки и на конкретное место в работе участника.
4. **Мягкое резюме** — сильные стороны, 2–3 главных направления для улучшения, общая зрелость работы.
   **Без жёсткого pass/fail** — решение и доработка за человеком.

## Формат фидбэка

Структурируй по осям. На каждую находку — **что, почему (принцип), как поправить**. В конце — мягкое резюме.

```
## Процесс
- ✅ <ось>: что сделано хорошо
- ⚠️ <ось>: что не продумано/неприменено — <принцип из справки> — корректировка: <как>
## Датасет
- ✅ / ⚠️ ...
## Резюме (мягкое)
Сильное: ... | Главное доработать: ... | Зрелость: ...
```

## Принципы скилла

- **Actionable, не назидательно.** Каждая претензия — с конкретной корректировкой.
- **Не правит за участника.** Reviewer = зеркало и советчик; правки вносит человек.
- **Гибкость по домену.** Не требуй «наших» типов/метрик — проверяй, что **фундаментальные принципы**
  применены осмысленно под **его** домен.
- **Честность об ограничениях.** Не предъявили процесс/валидацию — отметь, что часть осей не оценить.
- **Самопроверка допустима.** Нормально, если участник гоняет reviewer итеративно до приемлемого уровня —
  это часть обучения, а не обход проверки.
