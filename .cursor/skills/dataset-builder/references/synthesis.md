# Получение датасета: извлечение и синтез

Обезличенный конспект. Из понимания «что за данные» собираем **сами записи** «вход → ожидаемый
результат».

## Два пути, один подход

- **Извлечение** — вытащить готовые примеры из **реальных диалогов/логов**.
- **Синтез** — сгенерировать примеры **с нуля** из подготовленной базы знаний.

Вторая ось — **как размечают**: ручная / чистая синтетика / **гибрид**. Рабочий путь — гибрид: LLM
делает массу/черновик, человек и сильная модель валидируют. Поэтому в конце — обязательная
**валидация**.

## Общий инструмент: structured output и SGR

- **Structured output** — модель обязана вернуть ответ **строго по схеме** (описывают Pydantic-моделью);
  провайдер подкладывает схему, на выходе всегда валидный объект нужной формы.
- **SGR (Schema-Guided Reasoning)** — приём **поверх**: схемой **направляют само рассуждение** (какие
  шаги и в каком порядке). Не каждый structured output является SGR.

### Схема — это и есть промпт

| Элемент схемы | Что говорит модели |
|---|---|
| Имя класса | о какой задаче речь |
| Docstring | контекст и инструкции |
| Имена полей | смысл каждого шага |
| `Field(description=...)` | как именно заполнять поле |
| **Порядок полей** | в каком порядке рассуждать |
| Типы/`Literal`/диапазоны | какие значения допустимы |

Особенно важен **порядок**: сначала поле-рассуждение, потом поле-решение (иначе модель «ляпнет» вердикт
и будет его оправдывать).

```python
from pydantic import BaseModel, Field

class Review(BaseModel):
    """Сначала разбери, потом вынеси решение."""
    issues_found: str = Field(description="Какие проблемы видны")  # сначала рассуждаем
    needs_review: bool = Field(description="Нужна ли ручная проверка")  # потом решаем
```

Следствие: **системный промпт становится коротким** (формат описан схемой, дублировать не нужно;
промпт несёт только задачу и доменный контекст).

### Честный размен

✅ SGR даёт: воспроизводимость, тестируемость по полям, предсказуемую структуру.
⚠️ SGR **не гарантирует**: прирост качества ответов; жёсткая схема может его и снизить (модель тратит
«силы» на формат). Правило: structured output — где важна структура (сборка записей); свободный
текст — где важнее качество рассуждения (анализ уровня 1).

> Паттерны проектирования схем (каскад, routing, cycle) — в скиле
> [`schema-guided-reasoning`](../../schema-guided-reasoning/SKILL.md).

## Путь A — извлечение из реальных диалогов

Прямое продолжение уровня 2 анализа. **Группы из анализа становятся полями схемы** (например, категории
из таксономии → значения `Literal`-поля). Результат анализа не лежит мёртвым грузом — он **закодирован**
в схему.

### Схема извлечения = маленький конвейер рассуждения (каскад)

```python
from pydantic import BaseModel, Field
from typing import Literal

Category = Literal["cat_a", "cat_b", "cat_c"]  # ← из таксономии анализа

class Extraction(BaseModel):
    """Найти в окне диалога целевой случай."""
    # 1. Фильтры — быстрый отсев неподходящих окон
    has_user_question: bool = Field(description="Есть ли вопрос пользователя")
    is_on_topic: bool = Field(description="По теме ли")
    # 2. Извлекаем сам диалог
    conversation: list[dict] = Field(description="Реплики окна")
    # 3. Сначала РАССУЖДАЕМ, потом РЕШАЕМ
    reasoning: str = Field(description="Чего не хватает для точного ответа")
    category: Category | None = Field(description="Категория")
    # 4. Эталон — РЕАЛЬНЫЙ фрагмент из диалога (не выдумывать!)
    reference: str | None = Field(description="Дословно из диалога")
    # 5. Финал: годен ли кейс
    is_representative: bool = Field(description="Брать ли в датасет")
```

Правило, которое бережёт датасет: эталон должен быть **реальным** (дословным) фрагментом из диалога —
не выдумка.

### Скользящее окно по длинному диалогу

Диалог режут на пересекающиеся окна (например, 15 сообщений, перехлёст 2, шаг 13), каждое отдают модели
отдельно; в датасет попадают **только** окна с `is_representative=True`. Это фильтр от мусора.

### Несколько экстракторов под несколько групп

По каждой группе — свой экстрактор: одна механика (окно → схема → фильтр репрезентативности), разные
схемы и промпты.

### Кто запускает

Данные часто собирает **эксперт-предметник**: экспортирует логи, запускает скрипт (LLM делает
**черновик**), проверяет/правит/дополняет руками, отдаёт готовый датасет. Ответственность за качество —
на том, кто понимает домен.

## Путь B — синтез с нуля

Реальных диалогов нет — записи **генерируют** из подготовленной базы знаний (заранее нарезана на смысловые
фрагменты с метаданными: категория, аудитория).

### Типы датасетов — под каждую способность свой

Сначала решают, **что** проверяем (способность, не «вопросы»), и под это делают отдельный тип. Примеры
семейств (не обязательны — зависят от системы): простой поиск; сбор ответа из далёких фрагментов;
корректный отказ вне домена; «честный нет» на близкий к домену, но не покрытый вопрос; один вызов
инструмента; многоходовый диалог.

### Стратификация — но не всегда

Нужна там, где есть осмысленная **единица покрытия** (например, по категориям продуктов, по инструментам,
по типам сценариев). Где её нет (пары далёких фрагментов; негативы по доменам) — стратификация не нужна.

### Двухэтапная генерация (для покрытия режимов)

```
Инструмент/тема → Шаг 1: LLM придумывает 3–8 сценариев использования
               → Шаг 2: на каждый сценарий — вопрос пользователя + эталонный результат
```

Аналог «скользящего окна», только гонит по всем режимам инструмента/темы.

### Персоны (разнообразие)

~80% сгенерированных вопросов перефразируют под роль-персону, ~20% оставляют baseline (контроль). Правило:
роль меняет **стиль**, а не **смысл** (иначе сломается ожидаемый ответ). Воспроизводимость — фиксируй
`RANDOM_SEED`.

### Точность через схему + негативные инструкции

```python
# Literal фиксирует допустимое значение — модель физически не выдаст неправильное
name: Literal["target_tool"] = Field(description="Имя инструмента")
```

Плюс **негативные инструкции** в промпте («НЕ генерируй академические вопросы / то, на что отвечают из
одного фрагмента»).

### Инвариант multi-chunk

Если проверяем сбор ответа из нескольких фрагментов — берут **семантически далёкие** фрагменты (из разных
категорий), иначе многошаговость не проверится (система найдёт всё за один поиск).

### Agent-датасет неотделим от окружения

Эталонный вызов ссылается на конкретные ID (счёт, телефон…). Прогон надо вести на согласованном (mock)
окружении с теми же ID — иначе метрика ложно низкая. Это уже часть управления.

## Security-датасет (особый случай синтеза)

Для публичной системы — отдельно проверяем устойчивость к атакам. Собирают **тем же синтезом**; меняются
форма записи и метрика. Ключевое решение: **проверяем поведение под атакой, агностично от защиты** (не
важно, есть ли отдельный «гард») — на вход атака, смотрим итог: поддался / устоял.

Форма записи — ChatML, где вход = **атака** (одно-/многошаговая), в `metadata` — *ожидаемое безопасное
поведение* и *категория атаки*. Категории (+ benign-control как контроль ложных срабатываний):
`ignore_instructions`, `role_override`, `system_prompt_exfiltration`, `jailbreak`, `domain_manipulation`,
`indirect_injection` (инструкция спрятана **внутри** контента-данных), `invisible_payload` (инструкция в
невидимых Unicode-символах), `benign_control`. Метрика — **Security Accuracy** через LLM-судью (эталонного
текста нет; судья смотрит атаку + ответ + критерий «что считать поддачей»).

## Формат записи: ChatML

`input` / `expected_output` (+ произвольные поля в `metadata`) — максимально стандартный формат «роль +
содержимое». Любой датасет приводим к нему, нестандартное (ожидаемая категория, эталонный вызов, рубрика,
relevant_facts) → `metadata`. Мультимодальность: картинки кладут в `image_url` сообщения; датасет держат в
двух формах — пути к файлам (для человека) и base64 (для загрузки).

**▶ Стоп:** покажи человеку **выборку готовых записей**, прежде чем гнать весь объём.

## Валидация (три уровня)

1. **Автоматическая** — структурная (поля/типы), покрытие (все категории/роли, нет дублей), семантическая
   (например, фрагменты правда из разных категорий).
2. **Validation Sample** — горсть случайных + по одному на роль/категорию глазами: естественно ли звучит,
   сохранён ли смысл, корректен ли эталон.
3. **Прогон через сильную модель** — если **даже она** показывает плохой результат, проблема, скорее
   всего, **в датасете** (кривой эталон, противоречия), а не в системе.

Сквозной принцип: **лучше меньше, да лучше**.
