# Управление датасетом: пополнение и версионирование

Обезличенный конспект. Датасет — не разовая поделка: он **живёт вместе с агентом**. С ним происходят две
вещи: его **пополняют** (особенно кейсами, на которых агент ошибся в эксплуатации) и **версионируют**.

> **Граница.** Здесь — как пополнять и версионировать. Сам прогон, тюнинг метрик, red teaming — стадия
> Evaluation.

## Часть 1. Пополнение из эксплуатации

### Датасет растёт из провалов

Лучшие тест-кейсы **ловятся в бою**. Идея: не копить гигабайты логов, а на каждом запросе мерить метрики и
**целенаправленно разбирать сложные/провальные кейсы**.

```
агент в проде → сигнал провала (дизлайк/ошибка/жалоба) → отбор+разбор кейса
   → разметка (что было верно) → новая запись → ретест на новой версии → (новые слабые места) → ...
```

### Сигнал провала

- **Кнопки 👍 / 👎** под ответом → **score** к **трейсу** (булев, идемпотентный: повторное нажатие
  переключает, не плодит дубликаты). *Трейс* — записанный след одного запроса; *score* — оценка к трейсу.
- Остальное менее формально: сами замечаем странности, ловим ошибки, обратную связь приносит заказчик.

### Что делать с кейсом — ручные пути (на небольшом объёме)

1. **Эксперт ведёт таблицу** (Excel) по группам — ему привычнее, чем осваивать инструменты разметки;
   таблица становится исходником новых версий.
2. **Полуручной разбор трейсов** — экспорт трейса → разбор в паре с LLM (выручает записанный **reasoning**
   агента: видно *почему* ошибся, а не только *что* ответил) → решаем, в какой датасет.

### Системный способ — annotation queues

Ручные пути плохо масштабируются. Очередь аннотирования + **score config** (что и по какой шкале
оцениваем): кладёшь трейсы → размечают **несколько людей, в т.ч. нетехнических**, прямо в вебе → размеченный
трейс одной кнопкой → в датасет (с обратной ссылкой на трейс). Это целевой способ непрерывного улучшения:
провал в проде → очередь → разметка → датасет → ретест.

## Часть 2. Версионирование

### Версия = иммутабельный файл

- каждая версия — **отдельный файл** `v<NNN>_<дата>[_метка]` (напр. `v001_2026-02-15`,
  `v002_2026-03-02_customer`);
- файл версии **не редактируют** — правка = новая версия (иммутабельность как правило процесса);
- актуальная версия — последняя по сортировке имён;
- **файлы — источник правды**, система трекинга (Langfuse) — зеркало (заливается при старте бэкенда).
- форматы могут уживаться: **Excel** (заказчику привычнее) + **JSON** (команда).

### Folders-as-versions

Встроенное версионирование Langfuse правит item точечно, а нужны **снимки целиком**. Поэтому каждый
файл-версию заливают как **отдельный датасет** с именем `имя/версия` (напр. `rag_consult_qa/v001_2026-02-15`);
косая черта **группирует версии в одну папку** в UI → «папка = история версий».

### Идемпотентная загрузка

При старте проходим версии и заливаем **идемпотентно**: есть непустой датасет с таким именем — пропускаем,
иначе создаём. Следствие: новый файл-версия просто зальётся как новый датасет, уже залитые не трогаем.
Перезаливки «по содержимому» нет → **правка = новый файл с новым именем**.

### Грабли: бинарные вложения + git

Картинки тяжёлые/бинарные → `images/` уходит в `.gitignore` → на чистой машине датасет с картинками **не
поднимется** (файлов нет, загрузчик падает). Вывод: где есть бинарные вложения — **заранее реши, где они
живут** (git с исключениями, отдельное хранилище, артефакты сборки).

### Контраст: синтетика проще

Синтетика из базы знаний — фиксированные файлы без версий/папок/картинок. Полноценное версионирование
нужнее там, где источник **живой и пополняется**. Чем «живее» источник, тем нужнее дисциплина версий.

## Обезличивание

Реальные данные **обезличивай** — это часть процесса, а не формальность. Делается на источнике/при
извлечении; спорные остатки PII лучше исключать явно, помечая отсечение.
